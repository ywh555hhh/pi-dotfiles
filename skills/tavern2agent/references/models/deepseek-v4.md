# DeepSeek V4

模型特化建议。只在目标运行模型是 DeepSeek V4 或同类“user role 权重明显高于 system”的模型时使用。

## 核心差异

- system 服从度弱，长 system 更弱。
- user message 权重高，硬规则应靠近最新用户消息。
- 英文工具返回/英文 JSON 键名容易诱发 reasoning 语言漂移。
- 叙事模型会倾向继续写，不主动查工具。

## Prompt 分层

system 极短：身份 + 最高契约。

```md
你是《世界名》的 GM。
机械层来自工具；叙事层由你表达。未经工具确认的机械事实不存在。
```

每轮注入建议顺序：

```txt
参考上下文：世界摘要、工具入口、角色速查
玩家最新消息
硬规则：工具纪律、禁令、文风要求
```

实现方式按当前 pi API 调整；不要无脑复制旧 hook 模板。

## 全链路中文化

目标：减少英文 token 注入，不给 reasoning 漂移触发条件。

尽量中文化：

- data JSON 对模型可见的 key。
- 工具返回文本。
- 工具参数 description。
- GM prompt、规则、错误信息。

引擎内部可用英文；工具层映射成中文可见接口。不要做长期中英双轨兼容。

## 工具纪律

V4 叙事惯性强，工具 description 要硬：

```txt
必须调用：具体场景
严禁：凭记忆编造设定/价格/判定
职责：你把工具结果翻译成叙事，不创造机械结果
```

GM 规则里强调：

```txt
不调工具 = 该机械事实不存在。
```

## 文风控制

少给大段参考语料，多给可执行技巧：

- 优先级：最重要/其次/再次。
- 频率：每段 1 次、偶尔、大量、禁止。
- 句长：短句为主，长句只在高潮。
- 反例：不要这样写。

抽象词如“有文学性”“更有张力”效果差。

## XML / 标签

不要依赖标签名传义。V4 对 Markdown 标题更稳。

如果用标签，标签内也要写明内容是什么。

## 适用性

| 目标模型 | 建议 |
|---|---|
| DeepSeek V4 | 应用本文 |
| DeepSeek V3/R1 | 部分参考 |
| Claude/GPT | 通常不需要；按通用分层即可 |

## 限制

- reasoning 语言不能完全控制。
- pi 内置英文提示不可完全消除。
- 每轮注入有 token 成本，但 V4 长上下文和 cache 通常可接受。
